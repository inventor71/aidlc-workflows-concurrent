# /ai-dlc-refactor — AI-DLC 리팩토링 트랙

기존 구현을 **더 효율적이고 유지보수하기 좋은 구조로 재설계**하기 위한 커맨드.
AI-DLC 본 워크플로(feature delta 중심)가 다루지 못하는 "동작 보존 재설계"와
"의도 변경/기능 cut 승인 게이트"를 보강한다.

대상(범위): $ARGUMENTS
(비어 있으면 사용자에게 "어떤 모듈/파일/관심사를 재설계할지" 먼저 묻는다.)

---

## 0. 운영 원칙

- 이 커맨드는 **AI-DLC를 대체하지 않고 보강**한다. 본 룰셋의 audit 로깅·승인 게이트·
  content-validation 원칙을 그대로 따른다.
- 문서 산출물은 **`aidlc-docs/inception/refactor/<name>/`** 아래에만 둔다.
  애플리케이션 코드는 워크스페이스 루트(절대 `aidlc-docs/` 안 금지).
- 모든 사용자 입력/승인은 `aidlc-docs/audit.md`에 **append**로 기록(덮어쓰기 금지).
- 질문 파일이 필요하면 **한국어**로 작성한다(프로젝트 기본).
- 설계 승인 후 T1/T2 구현은 **자율 진행**한다(autonomy-in-construction). T3만 멈춘다.

---

## 1. 핵심 개념: 변경 분류 (Change Tier)

모든 변경 항목은 아래 3-tier 중 하나로 분류되어 `2-tier-ledger.md`에 기록된다.
**분류 없이 코드를 건드리지 않는다.**

| Tier | 정의 | 처리 |
|------|------|------|
| **T1 — 동작 보존** | 외부 관측 가능 동작·인터페이스·출력 동일. 내부 구조만 개선. | 설계 승인 후 **자율 진행** |
| **T2 — 안전한 확장** | 기존 동작 보존 + 새 경로/옵션 추가(superset). | **자율 진행 + 사후 보고** |
| **T3 — 의도 변경 / 기능 cut** | 기능 삭제, 의미 변경, backward-compat 포기, 계약(시그니처/스키마/출력) 축소. | **🛑 정지. 사용자 승인 전 코드 반영 금지** |

판단이 애매하면 **항상 더 높은 tier로 올린다**(보수적). "단순화를 위해 일부 기능 삭제/수정"은
거의 항상 **T3**임을 기억한다.

### T3 게이트 동작 (제안만 하고 정지)
- T3 후보는 ledger에 **"cut/변경 후보"로만 적고, 코드에는 절대 반영하지 않는다.**
- 각 T3 항목에는 다음을 함께 적는다:
  - 무엇을 cut/변경하려는가
  - **이유**(왜 backward-compat 유지가 복잡도를 키우는가 — 구체적 비용)
  - cut했을 때 얻는 것 / 잃는 것
  - 영향 받는 호출부·사용자·테스트
- T3 항목이 하나라도 있으면, 3단계(재설계) 진행 전 **사용자에게 일괄 제시하고 멈춘다.**
  사용자가 항목별로 "cut 승인 / 유지 / 보류"를 결정하기 전에는 다음으로 넘어가지 않는다.

---

## 2. 진행 단계

각 단계는 산출물을 만들고 **사용자 승인 후** 다음으로 간다(2-옵션: 변경 요청 / 계속).

### 단계 1 — Baseline + 특성화 테스트 (현재 동작 고정)
산출물: `aidlc-docs/inception/refactor/<name>/1-baseline.md` + 특성화 테스트 코드
- 대상 범위의 현재 구조 요약(핵심 파일·진입점·의존성).
- **보존해야 할 관측 가능 동작 목록**(외부 계약): 공개 함수 시그니처, CLI/출력 포맷,
  파일/DB 스키마, 부수효과, 에러 동작.
- 현재 테스트 커버리지 현황(있으면 어떤 경로를 덮는지).
- **특성화(characterization) 테스트 우선**: 재설계에 들어가기 전에, 보존할 동작을
  고정하는 테스트를 **먼저 작성/확인**한다. 기존 테스트가 그 경로를 이미 덮으면 그걸
  활용하고, 공백이 있으면 현재 동작을 그대로 캡처하는 테스트를 새로 추가한다.
  - 이 테스트는 "현재가 옳다"가 아니라 "현재가 이렇다"를 기록하는 것 — 리팩토링 중
    **before/after green**을 유지하는 안전망이다.
  - 단계 4 구현 내내 이 테스트를 계속 green으로 유지한다. red가 나면 그 변경은
    T1이 아니라 **T3(동작 변경)** 신호이므로 정지하고 ledger로 올린다.

### 단계 2 — Tier Ledger (분류 + cut 상의)
산출물: `aidlc-docs/inception/refactor/<name>/2-tier-ledger.md` (템플릿은 §3)
- 계획한 모든 변경을 항목으로 쪼개 T1/T2/T3 분류.
- **각 T1/T2 항목이 단계 1의 어느 특성화 테스트로 보호되는지** 매핑한다.
  보호하는 테스트가 없는 항목은 단계 1로 돌아가 테스트를 먼저 추가한 뒤 진행한다
  (특성화 테스트 우선 원칙).
- T3 항목이 있으면 → **여기서 정지**, 사용자에게 cut 안건 제시(객관식/표).
  사용자 결정을 ledger에 반영하고 audit.md에 기록한 뒤 단계 3으로.

### 단계 3 — Redesign (목표 구조 + 마이그레이션)
산출물: `aidlc-docs/inception/refactor/<name>/3-redesign.md`
- 목표 구조와 **동치성 논증**(T1 항목이 왜 동작을 보존하는지).
- 승인된 T3 변경의 새 동작 명세.
- 단계적 마이그레이션 순서(작은 단위, 각 단계 후 테스트 green 유지).

### 단계 4 — 구현 (AI-DLC Construction)
- 단계 1~3 승인 후, AI-DLC code-generation 흐름으로 진행.
- T1/T2: 자율 구현 + 매 단계 테스트 통과 확인. T2는 무엇을 추가했는지 사후 보고.
- T3: **승인된 항목만** 구현. 승인되지 않은 cut은 절대 손대지 않는다.
- 코드 요약 문서는 `aidlc-docs/construction/<unit>/code/`에.

## 3. `2-tier-ledger.md` 템플릿

```markdown
# Tier Ledger — <name>

범위: <대상 모듈/파일>
작성일: <YYYY-MM-DD>

## T1 — 동작 보존 (자율 진행)
| # | 변경 항목 | 보존되는 동작 | 보존 검증 방식 | 근거 |
|---|-----------|---------------|----------------|------|
| 1 | (예) X 모듈 함수 분리 | 공개 API 동일 | 기존 테스트로 충분 | 해당 경로 커버됨 |
| 2 | (예) Y 중복 제거 | 출력 동일 | 특성화 테스트 필요 | 기존 테스트 공백 |

## T2 — 안전한 확장 (자율 진행 + 사후 보고)
| # | 추가 항목 | 기존 동작 영향 | 보존 검증 방식 |
|---|-----------|----------------|----------------|
| 1 |           | 없음(superset) |                |

## T3 — 의도 변경 / 기능 cut (🛑 승인 필요)
| # | cut/변경 내용 | 이유(복잡도 비용) | 얻는 것 | 잃는 것 | 영향 범위 | 사용자 결정 |
|---|---------------|-------------------|---------|---------|-----------|-------------|
| 1 |               |                   |         |         |           | 승인/유지/보류 |

## 정지 지점
- [ ] T3 항목 사용자 제시 완료
- [ ] 사용자 결정 반영 + audit.md 기록 완료
```

---

## 4. 시작 절차

1. 범위($ARGUMENTS) 확인 — 없으면 사용자에게 질문.
2. `aidlc-docs/audit.md`에 이 호출을 append 기록.
3. 단계 1(Baseline)부터 시작. 각 단계 산출물 작성 후 승인 대기.
4. T3 발견 시 즉시 정지하고 사용자와 상의.
